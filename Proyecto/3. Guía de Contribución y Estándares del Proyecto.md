**Documento Vivo:** Esta guía establece las reglas de oro para escribir código, diseñar la interfaz y colaborar en el repositorio. Todo contribuidor debe leer y aplicar estas convenciones antes de abrir un Pull Request.

---

## 1. Convenciones de Código (Naming Conventions)

Para mantener una base de código coherente, utilizaremos las siguientes reglas de nomenclatura en todo el proyecto de TypeScript y Next.js.

* **Variables y Funciones:** `camelCase` (ej. `getUserData`, `isLoggedIn`, `taskCount`).
* **Componentes de React:** `PascalCase` (ej. `TaskCard.tsx`, `UserProfile.tsx`).
* **Interfaces y Tipos (TypeScript):** `PascalCase`. No usar el prefijo "I" (ej. `User`, `TaskProps`, en lugar de `IUser`).
* **Archivos y Carpetas:** `kebab-case` para utilidad y rutas de Next.js (ej. `utils/format-date.ts`, `app/dashboard-view`). *Excepción:* Los componentes de UI pueden llevar el nombre del componente (ej. `components/Button.tsx`).
* **Constantes Globales y Variables de Entorno:** `UPPER_SNAKE_CASE` (ej. `MAX_FILE_SIZE`, `NEXT_PUBLIC_API_URL`).

## 2. Flujo de Git y Mensajes de Commit

Adoptamos el estándar de **Conventional Commits** para que el historial del proyecto sea legible y predecible.

### Nomenclatura de Ramas (Branches)
* `feat/nombre-de-la-tarea` (Para nuevas funcionalidades. Ej: `feat/modal-login`)
* `fix/nombre-del-bug` (Para correcciones de errores. Ej: `fix/error-carga-imagen`)
* `refactor/nombre-del-modulo` (Para mejoras de código sin cambiar la funcionalidad)
* `docs/readme` (Para actualizaciones de documentación)

### Estructura del Commit
Cada commit debe empezar con un prefijo que indique su propósito, seguido de una descripción breve en minúsculas y en modo imperativo.
* `feat: agrega el componente de barra de busqueda`
* `fix: corrige el desbordamiento de texto en las tarjetas`
* `chore: actualiza las dependencias de prisma`

## 3. Sistema de Diseño (UI/UX)

Para evitar tener 50 tonos de gris diferentes, estandarizamos nuestro diseño centralizándolo en el archivo `tailwind.config.ts`. Solo se deben usar estas clases.

### Paleta de Colores

| Concepto | Clase Tailwind | Código Hexadecimal | Uso principal |
| :--- | :--- | :--- | :--- |
| **Primario** | `bg-primary` | `#2563EB` (Blue 600) | Botones principales, enlaces, elementos destacados. |
| **Secundario** | `bg-secondary` | `#4F46E5` (Indigo 600) | Acciones secundarias, iconos. |
| **Fondo Base** | `bg-background` | `#F8FAFC` (Slate 50) | Color de fondo de la aplicación entera. |
| **Superficie** | `bg-surface` | `#FFFFFF` (White) | Fondo de tarjetas, modales y menús desplegables. |
| **Texto Principal** | `text-slate-900` | `#0F172A` | Títulos y texto de alto contraste. |
| **Texto Secundario** | `text-slate-500` | `#64748B` | Párrafos, subtítulos y placeholders. |
| **Éxito (Success)** | `bg-green-500` | `#22C55E` | Alertas de éxito, estados "Completado". |
| **Peligro (Danger)**| `bg-red-500` | `#EF4444` | Botones de eliminar, alertas de error. |

### Tipografía y Espaciado
* **Fuente Principal:** `Inter` (Sans-serif). Optimizada para legibilidad en interfaces.
* **Títulos (H1, H2):** Usar pesos `font-bold` o `font-semibold`.
* **Cuerpo de texto:** Usar tamaño `text-base` o `text-sm` con peso `font-normal`.
* **Espaciado (Margin/Padding):** Usar siempre la escala por defecto de Tailwind (múltiplos de 4px). Ej: `p-4` (16px), `mb-8` (32px).
* **Bordes Redondeados (Border Radius):** Usar `rounded-md` para botones e inputs, y `rounded-lg` para tarjetas o modales.

## 4. Arquitectura Next.js y Componentes

Para aprovechar al máximo el App Router de Next.js, seguiremos estas reglas estructurales:

* **Server Components por defecto:** Asumir que todo componente es de servidor. Solo agregar `"use client"` en la primera línea si el componente necesita interactividad (`useState`, `onClick`, `useEffect`).
* **Componentes Tontos vs Inteligentes:** Separar la lógica de la vista. Si un componente de UI (ej. un Botón) se usa en más de dos lugares, debe ir a la carpeta `/components/ui` y no debe hacer peticiones a la base de datos directamente.
* **Orden de Imports:** Para mantener los archivos limpios, los `import` deben seguir este orden estricto (separados por una línea en blanco):
  1. Librerías externas (ej. `react`, `next/link`).
  2. Componentes y utilidades internas usando alias (ej. `@/components/Button`).
  3. Archivos de estilos o recursos multimedia.

> 💡 **Regla de Oro:** "Boy Scout Rule".
> Deja el código un poco más limpio de como lo encontraste. Si ves una variable sin usar o un tipo `any` mientras trabajas en tu tarea, arréglalo o repórtalo.

---

## 5. Calidad de Código y Testing Manual

* **Cero Consola:** No dejar `console.log()` en el código de producción. Utilizarlos solo para depurar en local y borrarlos antes del commit.
* **Tipado Estricto:** Prohibido el uso de `any` en TypeScript. Si no conoces el tipo exacto, usa `unknown` y valídalo, o crea la interfaz correspondiente.
* **Prettier y ESLint:** Todo el código debe formatearse automáticamente al guardar. Si ESLint marca un error (subrayado rojo en VSCode), el Pull Request no será aprobado hasta que se corrija.
