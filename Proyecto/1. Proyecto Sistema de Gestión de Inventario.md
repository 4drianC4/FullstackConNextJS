# Metodología: Desarrollo por "Verticales" (Slices) y Code Review

## 1. Visión General del Proyecto
Construiremos desde cero un **Sistema de Gestión de Inventario con Tienda Online Integrada**. El sistema constará de dos grandes áreas interconectadas: un panel de administración (Backoffice) para controlar el stock, productos y órdenes, y una interfaz pública (E-commerce) donde los clientes finales podrán navegar por el catálogo, añadir productos a un carrito y realizar pedidos.

El objetivo principal es simular un entorno de desarrollo profesional real. Todo el código se integrará en un único repositorio centralizado (Monorepo o Repositorio Unificado) en GitHub. Cada avance se trabajará en ramas independientes y deberá ser revisado, comentado y aprobado mediante *Pull Requests* antes de pasar a la rama principal.

## 2. Stack Tecnológico
Para garantizar homogeneidad en el aprendizaje y agilidad en el despliegue, utilizaremos un stack moderno basado íntegramente en JavaScript/TypeScript:

* **Frontend (Cliente):** Next.js (App Router), React, Tailwind CSS, Zustand (Manejo de estado global), React Hook Form + Zod (Validación de formularios).
* **Backend (Servidor):** Next.js (Route Handlers / API Routes), Node.js.
* **Base de Datos y ORM:** PostgreSQL (alojado en Supabase o Neon), Prisma ORM.
* **Infraestructura y Herramientas:** Vercel (CI/CD y Hosting), GitHub (Control de versiones), Git, ESLint / Prettier.

## 3. Arquitectura y Metodología de Trabajo
Para evitar que el incumplimiento o retraso de un estudiante bloquee el progreso de los demás, aplicaremos el enfoque de **Desarrollo Vertical (Vertical Slices)**. Cada desarrollador será propietario de una funcionalidad completa de extremo a extremo (Fullstack): diseñará sus modelos en la base de datos, creará sus propios endpoints en la API y maquetará la interfaz correspondiente.

### El Flujo de Vida de una Tarea (Feature Branch Workflow)
1.  **Asignación:** El desarrollador recibe su módulo asignado con sus requerimientos mínimos.
2.  **Diseño DB (Aislado):** Modifica el archivo `schema.prisma` agregando únicamente las tablas o campos necesarios para su característica.
3.  **Backend:** Construye y prueba de forma aislada sus endpoints RESTful (POST, GET, PUT, DELETE) utilizando herramientas como Thunder Client o Postman.
4.  **Frontend:** Diseña y conecta los componentes visuales con la API que acaba de construir, gestionando los estados necesarios.
5.  **Integración:** Abre un *Pull Request (PR)* en GitHub. El código solo se unirá a la rama `main` tras la revisión de código (*Code Review*) y aprobación del Tech Lead (Instructor) y de al menos un compañero.

## 4. Módulos de Desarrollo
El proyecto se compone de 8 módulos diseñados con una arquitectura desacoplada. Los módulos **1 al 6 representan el núcleo esencial** de la aplicación. Los módulos **7 y 8 actúan como extensiones independientes**, garantizando que si el curso inicia finalmente con 6 alumnos, el sistema siga siendo completamente funcional y coherente.

### Módulo 1: Autenticación, Usuarios y Roles (Núcleo)
* **Descripción:** Gestión de sesiones de usuarios y control de acceso.
* **Responsabilidades Fullstack:**
    * Modelos en la BD para `User`, `Session` y `Role` (Admin, Cliente, Personal).
    * Endpoints para registro, inicio de sesión y cierre de sesión.
    * Implementación de Middleware de Next.js para proteger rutas del panel de administración y rutas del cliente (historial de compras).
* **Punto de desacoplamiento:** Proporciona las funciones de sesión global (`useAuth`) y los IDs de usuario que los demás módulos consumirán de manera pasiva.

### Módulo 2: Catálogo de Productos y Categorías - Vista Admin (Núcleo)
* **Descripción:** Panel de administración para la gestión de la oferta comercial.
* **Responsabilidades Fullstack:**
    * Modelos de `Product` y `Category` (Relación uno-a-muchos).
    * CRUD completo (Crear, Leer, Actualizar, Eliminar) de categorías y productos.
    * Formularios validados con Zod para la carga de datos (Nombre, descripción, precio, SKU, imágenes mediante URLs).
* **Punto de desacoplamiento:** Si este módulo se retrasa, otros módulos pueden usar un archivo de datos simulados (*mock data*) respetando el formato del modelo de Prisma.

### Módulo 3: Control de Stock y Movimientos de Inventario (Núcleo)
* **Descripción:** Sistema interno para controlar la disponibilidad real de la mercancía.
* **Responsabilidades Fullstack:**
    * Modelos de `InventoryAdjustment` o logs de movimientos (Entradas por proveedor, salidas manuales, mermas).
    * Interfaz de administración para modificar el stock de un producto existente y ver el historial de cambios de unidades.
    * Lógica para activar alertas visuales en el panel cuando un producto cruce un umbral de "Stock Mínimo".
* **Punto de desacoplamiento:** Se conecta al producto mediante su ID, pero el flujo operativo de registro es totalmente independiente del catálogo visual.

### Módulo 4: Tienda Online - Catálogo y Filtros Públicos (Núcleo)
* **Descripción:** La cara pública del negocio orientada al cliente final.
* **Responsabilidades Fullstack:**
    * Endpoints de consulta pública con soporte para paginación y ordenamiento (por precio, novedades).
    * Interfaz pública responsiva que renderice las tarjetas de los productos activos.
    * Barra de búsqueda de productos y barra lateral de filtros funcionales por categoría y rangos de precio.
* **Punto de desacoplamiento:** Es puramente de lectura de la tabla de productos. No escribe en la base de datos del inventario de forma directa.

### Módulo 5: Carrito de Compras y Creación de Pedidos (Core)
* **Descripción:** Gestión de la intención de compra del cliente y persistencia local.
* **Responsabilidades Fullstack:**
    * Manejo de estado global con Zustand para el carrito (Añadir, remover, cambiar cantidades, calcular totales e impuestos en caliente).
    * Interfaz de la página del carrito y formulario de Checkout (Datos de envío y método de pago simulado).
    * Endpoint para procesar el Checkout que reciba el carrito y cree un registro inicial en la tabla `Order` en estado "Pendiente".
* **Punto de desacoplamiento:** Guarda la orden en la base de datos de manera aislada. No gestiona el cambio posterior de estados ni la lógica compleja de despacho.

### Módulo 6: Gestión de Órdenes y Ventas - Panel Admin (Core)
* **Descripción:** El flujo de trabajo posterior a la venta y actualización automática del negocio.
* **Responsabilidades Fullstack:**
    * Modelos de `Order` y `OrderItem` conectados a los usuarios y productos.
    * Panel de administración para visualizar todas las órdenes realizadas, filtradas por estado (Pendiente, Pagado, Enviado, Cancelado).
    * Mecanismo de cambio de estado: Al marcar una orden como "Pagada/Confirmada", de manera atómica a nivel de backend, se descuentan las unidades correspondientes del Módulo de Inventario.
* **Punto de desacoplamiento:** Depende de la existencia de registros en la tabla `Order`. Si el módulo 5 se retrasa, el administrador puede crear órdenes de prueba manualmente mediante un script o interfaz básica para validar el flujo.

### Módulo 7: Sistema de Cupones y Descuentos (Extensión)
* **Descripción:** Módulo de marketing para incentivar compras mediante reglas de precios.
* **Responsabilidades Fullstack:**
    * Modelo `Coupon` (Código único, tipo de descuento: porcentaje/fijo, valor, fecha de expiración, estado activo).
    * Panel de administración para crear, listar y desactivar cupones de descuento.
    * Endpoint de validación: Recibe un string, verifica si es válido y retorna el descuento aplicable. Se integra opcionalmente como un input extra en el Módulo 5 (Checkout).
* **Punto de desacoplamiento:** Es completamente opcional. Si no se completa, el input de cupones simplemente se oculta en el carrito y la tienda sigue cobrando el total normal sin romper la estructura.

### Módulo 8: Panel de Estadísticas y Reportes Analíticos (Extensión)
* **Descripción:** Dashboard visual de analítica de negocio para la toma de decisiones.
* **Responsabilidades Fullstack:**
    * Endpoints analíticos que ejecuten agregaciones en la base de datos (Suma de ingresos totales, conteo de órdenes mensuales, productos más vendidos).
    * Interfaz con gráficos estadísticos (usando librerías ligeras como `recharts` o componentes puros CSS/SVG) para mostrar el rendimiento de las ventas.
    * Mecanismo para exportar el estado actual del inventario crítico a un formato plano imprimible o CSV básico.
* **Punto de desacoplamiento:** Es un módulo de consumo 100% pasivo (lectura). No altera ninguna tabla ni interfiere con los flujos de compra o administración diarios.
### Módulo 9: Reseñas y Calificaciones de Productos (Extensión)

**Descripción:** Sistema para que los usuarios registrados dejen feedback público sobre los productos que han comprado o visualizado. **Responsabilidades Fullstack:**

- Modelos en la BD para `Review` (relacionado con `User` y `Product`, incluyendo puntuación de 1 a 5 y comentario en texto).
    
- Endpoints (API) para crear una reseña, listar reseñas de un producto específico y calcular el promedio de calificación.
    
- Interfaz pública que incluya estrellas de calificación en las tarjetas del Módulo 4 y una sección de comentarios en la vista de detalle del producto.
    
- **Punto de desacoplamiento:** Es información puramente complementaria. Si no está listo, la tienda simplemente no muestra las estrellas de calificación, pero el catálogo y las compras siguen funcionando normalmente.
    

### Módulo 10: Lista de Deseos (Wishlist) (Extensión)

**Descripción:** Funcionalidad para que los clientes guarden productos que les interesan para futuras compras. **Responsabilidades Fullstack:**

- Modelo en la BD para `WishlistItem` (relacionando `User` y `Product`).
    
- Endpoints para añadir/remover productos de la lista de deseos y consultar la lista completa del usuario activo.
    
- Interfaz con un botón de "corazón" en los productos del catálogo y una página privada en el frontend donde el usuario vea sus productos guardados.
    
- **Punto de desacoplamiento:** Es paralelo al Módulo 5 (Carrito). No afecta la compra actual y los estados de Zustand pueden manejarse de forma completamente aislada.
    

### Módulo 11: Gestión de Proveedores (Extensión)

**Descripción:** Panel administrativo para llevar el directorio de los suministradores de mercancía. **Responsabilidades Fullstack:**

- Modelo en la BD para `Supplier` (Nombre de empresa, contacto, email, teléfono, dirección).
    
- Endpoints CRUD para que el administrador gestione esta libreta de contactos comerciales.
    
- Interfaz en el Backoffice (Panel Admin) con formularios validados en Zod para registrar y listar proveedores.
    
- **Punto de desacoplamiento:** Inicialmente actúa como una libreta de direcciones independiente en el panel admin. En el futuro podría conectarse al Módulo 3 (Stock) para registrar entradas por proveedor, pero puede existir perfectamente por sí solo.
    

### Módulo 12: Perfil de Usuario y Libreta de Direcciones (Extensión)

**Descripción:** Panel de autoservicio para el cliente final donde puede gestionar sus datos personales y múltiples direcciones de envío. **Responsabilidades Fullstack:**

- Modelo en la BD para `Address` (relacionado a `User`, con campos de calle, ciudad, código postal y un booleano `isDefault`).
    
- Endpoints para crear, editar, eliminar y listar las direcciones de un usuario.
    
- Interfaz en el frontend (Mi Cuenta) para gestionar el perfil y las direcciones.
    
- **Punto de desacoplamiento:** El Módulo 5 (Checkout) puede requerir direcciones escritas manualmente si este módulo falla. Si este módulo se completa con éxito, el estudiante del Módulo 5 podrá consumir un endpoint para autocompletar la dirección del cliente.
    

### Módulo 13: Sistema de Soporte y Contacto (Tickets)

**Descripción:** Canal de comunicación directo entre los clientes y los administradores de la tienda. 
**Responsabilidades Fullstack:**

- Modelo en la BD para `SupportTicket` (relacionado a `User`, asunto, mensaje, estado: abierto/cerrado, y respuesta del admin).
    
- Endpoints para que el cliente cree un ticket y para que el administrador los lea y responda.
    
- Interfaz pública con formulario de contacto para el cliente y una vista en el Backoffice para que el administrador gestione la bandeja de entrada.
    
- **Punto de desacoplamiento:** Es un sistema de comunicación totalmente aislado del flujo comercial. No interactúa con productos, órdenes ni inventario.

## 5. Criterios de Éxito y Calificación
Para asegurar el correcto funcionamiento del ecosistema integrado, cada módulo individual será evaluado bajo las siguientes directrices puntuales:

1.  **Funcionalidad en Producción:** La característica asignada debe operar correctamente en el entorno desplegado en Vercel. No deben existir errores críticos en la consola del navegador ni respuestas con código de estado HTTP 500 en el servidor.
2.  **Calidad y Robustez del Código:** Ausencia de variables declaradas y no utilizadas. El tipado en TypeScript debe ser estricto, restringiendo por completo el uso del tipo genérico `any`. Todo el código debe pasar las reglas de formateo estipuladas por ESLint y Prettier.
3.  **Contrato de Datos (Mocking):** En caso de dependencias entre módulos, el estudiante debe demostrar haber acordado y documentado la interfaz de datos (*payloads* JSON) durante la primera semana, usando datos simulados si su contraparte presenta retrasos.
4.  **Colaboración y Code Review:** Participación activa en el flujo de trabajo de Git. Cada estudiante tiene la obligación de revisar, probar localmente y comentar de manera constructiva un mínimo de 2 *Pull Requests* de sus compañeros de clase antes de que el instructor realice el merge definitivo.
